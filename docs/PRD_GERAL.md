## Terms (for non-technical readers)

| Term | Meaning |
|---|---|
| **dApp** | Decentralized application — a web interface that interacts with a blockchain program via a wallet |
| **Wallet adapter** | Library that connects the browser wallet (Phantom, Solflare, Backpack) to the dApp |
| **Wallet standard** | A protocol where wallets self-register in the browser — the dApp auto-detects them without importing wallet-specific code |
| **PDA derivation** | Computing the on-chain storage address from known seeds (e.g., `["tree", creator, mint, campaignId]`). Frontend must compute these identically to the smart contract |
| **IDL** | Interface Description Language — machine-readable description of the smart contract's instructions, accounts, and errors. Generated by `anchor build` |
| **Merkle proof** | A short cryptographic proof (array of 32-byte hashes) that proves a recipient is in the campaign's beneficiary list, without storing the full list on-chain |
| **RPC endpoint** | Network URL that the frontend uses to communicate with Solana (read accounts, send transactions) |
| **ATA (Associated Token Account)** | Deterministic token account for a given wallet + mint pair. Each user has one ATA per token type |

## §1 Problem Statement

### 1.1 Protocol without UI = unusable for target users

Mancer's on-chain protocol (Lana's scope) solves the cost problem — 0.005 SOL flat cost for any campaign size vs. $1,990–$11,730 for competitors. But DAOs, project teams, and community managers are not CLI users. Without a frontend:

- Campaign creation requires building Merkle trees via command line
- Recipients must construct and submit `claim` transactions manually
- No visibility into vesting progress, unlock schedules, or campaign status
- Admin operations (pause, cancel, root rotation) require raw instruction calls

**Every competitor with market traction (Streamflow, Zebec) has a web UI.** Jito's Merkle distributor — the closest technical comparison — has no UI, which limits its adoption to developer-only users.

### 1.2 Frontend as the accessibility layer

The frontend transforms the protocol from a developer tool into a product:

| Without frontend | With frontend |
|---|---|
| Creator: build tree CLI → construct tx → sign → submit | Creator: upload CSV → review → click "Create" |
| Recipient: fetch proof from IPFS → build claim ix → sign | Recipient: connect wallet → see balance → click "Claim" |
| Admin: raw instruction calls for pause/cancel/rotate | Admin: dashboard buttons with confirmation modals |
| Status: `solana account` CLI queries | Status: real-time dashboard with charts |

### 1.3 Competitive positioning

| Feature | Streamflow UI | Zebec UI | Mancer UI (target) |
|---|---|---|---|
| Campaign creation | Per-stream form | Per-stream form | **Bulk CSV upload → Merkle compression** |
| Cost per 10K users | ~$3,720 | ~$11,730 | **~$0.42** |
| Vesting types | Linear only | Linear + milestone | **Cliff + linear + milestone (mixed)** |
| Real-time unlock | Yes | Yes | Yes |
| Clawback UI | Limited | Limited | **Campaign-wide + per-recipient** |
| Mobile responsive | Partial | Partial | **Full responsive** |

## §2 User Personas

### P1 — Campaign Creator (project team / DAO operator)

**Profile:** Non-technical or semi-technical. Manages token distribution for a DAO, gaming project, or DeFi protocol. Has a spreadsheet of recipients + amounts.

**Goals:**
- Create vesting campaign from a CSV of recipients
- Fund the campaign in one transaction
- Monitor claim progress
- Cancel campaign or remove specific recipients if needed

**Pain points:**
- Does not know CLI tools
- Cannot construct Merkle trees manually
- Needs confirmation before irreversible actions (fund, cancel)

### P2 — Recipient (beneficiary / contributor / investor)

**Profile:** End user with a Solana wallet. Received notification that tokens are vesting for them.

**Goals:**
- Connect wallet, see how much is vested
- Claim unlocked tokens with one click
- Track vesting schedule and upcoming unlocks

**Pain points:**
- Should not need to understand Merkle proofs
- Wants clear feedback on transaction status
- Needs to know exactly when tokens unlock

### P3 — Admin (cancel authority / pause authority)

**Profile:** Project lead or multisig signer. Has elevated permissions on a campaign.

**Goals:**
- Pause/unpause campaigns in emergencies
- Cancel campaigns and manage grace period
- Rotate Merkle root to remove/update recipients

**Pain points:**
- Destructive actions must have clear warnings
- Needs to understand consequences before acting

## §3 User Stories & Flows

### US-1: Create Campaign
```
Creator connects wallet
  → Uploads CSV (wallet, amount, schedule type, dates)
  → Frontend builds Merkle tree + computes root
  → Frontend pins leaves to IPFS/Pinata
  → Creator reviews: total supply, leaf count, root hash
  → Creator clicks "Create Campaign"
  → Frontend calls create_campaign instruction
  → Creator clicks "Fund Campaign"
  → Frontend calls fund_campaign instruction
  → Dashboard shows new campaign as "Active"
```

### US-2: Claim Tokens
```
Recipient connects wallet
  → Frontend fetches all campaigns where wallet is beneficiary
  → For each campaign: fetch proof from IPFS, compute vested amount
  → Recipient sees: total allocation, vested, claimed, claimable
  → Recipient clicks "Claim"
  → Frontend calls claim instruction with leaf + proof
  → Balance updates, transaction confirmed
```

### US-3: View Campaign Dashboard
```
Any user connects wallet (or views public campaign)
  → Dashboard shows: campaign metadata, total supply, claimed %, time remaining
  → Progress bar / chart for vesting curve
  → List of recent claim events
  → Campaign status: Active / Paused / Cancelled / Completed
```

### US-4: Cancel Campaign (Admin)
```
Admin connects wallet
  → Navigates to campaign admin panel
  → Clicks "Cancel Campaign"
  → Frontend shows warning: "This will freeze vesting. Recipients have 7 days to claim earned tokens."
  → Admin confirms
  → Frontend calls cancel_campaign instruction
  → Dashboard shows "Cancelled" status + grace period countdown
```

### US-5: Withdraw Unvested (Admin, post-grace)
```
Admin sees grace period expired on cancelled campaign
  → "Withdraw Unvested" button becomes active
  → Admin clicks, confirms
  → Frontend calls withdraw_unvested instruction
  → Remaining tokens returned to creator wallet
```

### US-6: Update Root / Remove Recipient (Admin)
```
Admin uploads new CSV without removed recipient
  → Frontend builds new Merkle tree
  → Frontend pins new proof set to IPFS
  → Admin reviews diff: who was removed, who was added
  → Admin clicks "Update Root"
  → Frontend calls update_root instruction
  → Old proofs for removed recipients are invalidated
```

## §4 Feature Requirements

Six features mapping to Lana's PRD §3, from the frontend perspective.

### F1 — Campaign Creation UI

**Priority:** P0  
**Maps to:** PRD_LANA F1 (Bulk Send)

**What the frontend must do:**
1. CSV upload and parsing — validate wallet addresses, amounts, schedule types, dates
2. Build Merkle tree client-side using `apps/web/src/lib/merkle/builder.ts`
3. Pin leaves to IPFS/Pinata for recipient proof retrieval
4. Render preview: total supply, leaf count, root hash, cost estimate
5. Call `createCampaign` instruction via Anchor
6. Call `fundCampaign` instruction after campaign creation
7. Display transaction confirmation with explorer links

**Input validation:**
- Wallet addresses: valid base58, 32–44 characters
- Amounts: positive integers, total matches sum of leaves
- Dates: start < cliff ≤ end, all in the future
- Release type: 0 (Cliff), 1 (Linear), 2 (Milestone)
- Duplicate wallet detection within same campaign

### F2 — Claim UI

**Priority:** P0  
**Maps to:** PRD_LANA F2 (Claim)

**What the frontend must do:**
1. On wallet connect, fetch all campaigns where connected wallet is a beneficiary
2. For each campaign, fetch Merkle proof from IPFS
3. Compute vested amount client-side: `unlocked = total * (elapsed / duration)`
4. Display: total allocation, vested, already claimed, claimable now
5. "Claim" button → call `claim` instruction with leaf + proof
6. Handle `CampaignPaused` error → show "Campaign is paused" message
7. Subscribe to `Claimed` event → refresh balances via TanStack Query invalidation

**Schedule display by type:**
| Type | UI |
|---|---|
| Cliff (0) | "Unlocks X tokens on [cliff date]" |
| Linear (1) | Progress bar: current % unlocked, next unlock timestamp |
| Milestone (2) | Checklist of milestones, each with date and status |

### F3 — Campaign Dashboard / Event Feed

**Priority:** P0  
**Maps to:** PRD_LANA F3 (Events)

**What the frontend must do:**
1. Campaign overview: metadata, total supply, claimed %, status badge
2. Vesting curve visualization (chart or progress bar)
3. Recent events feed using Anchor `addEventListener`
4. Campaign status display: Active / Paused / Cancelled / Completed / Grace Period

### F4 — Multi-Schedule Display

**Priority:** P1  
**Maps to:** PRD_LANA F4 (Customize Vesting)

**What the frontend must do:**
1. Show correct UI per `release_type` (cliff, linear, milestone)
2. Handle mixed campaigns — one campaign may have all three types
3. Milestone bitmap visualization (which milestones are claimed)

### F5 — Cancel & Grace Period UI

**Priority:** P1  
**Maps to:** PRD_LANA F5 (Clawback)

**What the frontend must do:**
1. "Cancel Campaign" button with destructive action warning
2. Grace period countdown timer (7 days)
3. "Withdraw Unvested" button — disabled until grace period expires
4. Visual indicator: campaign frozen, no new vesting accrual

### F6 — Root Rotation UI

**Priority:** P2  
**Maps to:** PRD_LANA F6 (Per-recipient Clawback)

**What the frontend must do:**
1. Upload new CSV (modified recipient list)
2. Build new Merkle tree, show diff (added/removed recipients)
3. Pin new proof set to IPFS
4. Call `updateRoot` instruction
5. Warn: "Removed recipients will no longer be able to claim"

### 5.3 Wallet strategy

**Wallet Standard auto-detect** — `wallets={[]}` in `WalletProvider`.

Rationale: `@solana/wallet-adapter-wallets` bundles 40+ adapters including several with React 16/17 peer dependencies. React 19 + that package = dependency chaos. Phantom, Solflare, and Backpack all implement the wallet standard protocol, so the adapter library is unnecessary — they auto-detect in the browser.

Supported wallets (auto-detected):
- Phantom
- Solflare
- Backpack
- Any wallet implementing the wallet standard protocol

### 5.4 State management

| Package | Version | Role |
|---|---|---|
| **TanStack React Query** | 5.67 | Server/chain state: account data, balances, campaign info. Handles caching, refetch, invalidation |
| **Zustand** | 5.0 | Client-only state: selected campaign, UI modals, form state. Lightweight, no boilerplate |

**Design decision (from Week 2 architecture §8.2):** Chain data lives in TanStack Query (auto-refetch on Solana events). UI-only state lives in Zustand. No Redux — overkill for this scope.

## §7 Non-functional Requirements

### NFR-1 — Performance

**Requirement:** First Contentful Paint (FCP) < 2s on 4G connection.  
**How:** Next.js SSR for initial render, lazy-load wallet adapter, code-split per route.  
**Verification:** Lighthouse score ≥ 85 on `/` route.

### NFR-2 — Accessibility

**Requirement:** WCAG 2.1 Level AA compliance for core flows (connect, create, claim).  
**How:** Semantic HTML, ARIA labels on interactive elements, keyboard navigation, color contrast ≥ 4.5:1.  
**Verification:** axe-core audit with 0 critical violations.

### NFR-3 — Responsive design

**Requirement:** Usable on viewports from 375px (iPhone SE) to 2560px (ultrawide).  
**How:** Tailwind responsive utilities, mobile-first design, no horizontal scroll.  
**Verification:** FAC8.

### NFR-4 — Browser support

**Requirement:** Latest 2 versions of Chrome, Firefox, Safari, Edge.  
**How:** Next.js handles transpilation. No browser-specific APIs without fallback.

### NFR-5 — Error resilience

**Requirement:** Every user-facing error includes: what happened, what to do next.  
**How:** Map Anchor error codes (6000–6027) to human-readable messages. Show tx explorer link on failure.  
**Verification:** Error mapping covers all 30 `VestingError` variants.

### NFR-6 — Security

**Requirement:** See `docs/SECURITY_GERAL.md` for full threat model.  
**Summary:** No private keys in client code, XSS prevention on chain-derived data, input validation before tx submission, CSP headers.
